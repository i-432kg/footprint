| 1 | 依存と責務分離 / 保守性 | `src/main/java/jp/i432kg/footprint/application/command/PostCommandService.java` | Medium | `PostCommandService` は画像をストレージへ保存した後、メタデータ抽出失敗や DB 永続化失敗が起きても保存済みオブジェクトを補償削除しない。そのためアプリケーション例外へ変換して終了した場合に orphan 画像が残る。 | `ImageStorage` に削除ポートを追加するか、アプリケーションサービスで補償処理を入れる。少なくとも `imageMetadataExtractor.extract(...)` と `postRepository.savePost(...)` の失敗時には保存済みオブジェクトの扱いを明示する。 | クローズ | `ImageStorage.delete(...)` が追加され、後続失敗時は cleanup を試行し、cleanup 失敗もログに残すようになった。 |
| 2 | 不変条件・境界値 / 一貫性 | `src/main/java/jp/i432kg/footprint/application/query/PostQueryService.java`<br>`src/main/java/jp/i432kg/footprint/application/query/ReplyQueryService.java` | Medium | `application.query` は `@NullMarked` 配下なのに、ページング用 `lastId` 引数が non-null シグネチャのままになっている。一方で実際の呼び出し側は `@RequestParam(required = false)` から `null` を渡し、実装・MyBatis mapper 側も `@Nullable` を前提にしているため、アプリケーション層の公開契約と実運用が食い違っている。 | `lastId` を `@Nullable` にするか、`Optional` など明示的なページング型へ寄せる。application interface、実装、呼び出し側の nullability 契約を一致させる。 | クローズ | `lastId` は application interface / 実装ともに `@Nullable` に揃えた。 |
| 3 | 一貫性 / 不変条件・境界値 | `src/main/java/jp/i432kg/footprint/application/query/model/PostSummary.java`<br>`src/main/java/jp/i432kg/footprint/application/query/model/ReplySummary.java`<br>`src/main/java/jp/i432kg/footprint/application/query/model/LocationSummary.java` | Medium | `application.query.model` は `@NullMarked` 配下だが、実データとして `null` を取りうる項目が non-null のまま定義されている。たとえば `PostSummary.location` は位置情報なし投稿で `null` を取りうるし、`ReplySummary.parentReplyId` はルート返信で `null` になる。実際に presentation mapper 側でも `Optional.ofNullable(...)` で防御しており、アプリケーション層の nullness 契約が崩れている。 | nullable な query model フィールドには `@Nullable` を明示するか、`hasLocation` と `location` の組、親返信あり/なしを表す専用型へ整理する。`application.query.model` を `@NullMarked` にする前提と実データ形状を一致させる。 | クローズ | DTO は増やさず nullable 明示で揃える方針を ADR 化し、必要フィールドへ `@Nullable` を付与した。 |
