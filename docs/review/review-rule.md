# レビューの運用ルール

## 優先度の基準
- **High**: リリース前に必ず修正が必要
- **Medium**: できればリリース前に修正したい
- **Low**: 後続対応でも許容可能

## No. の採番ルール
- 通し番号で管理する
- 再レビュー時も同一指摘は同じ No. を使う
- 新規指摘のみ新しい No. を採番する

## 対応状況の更新ルール
- 修正着手時: 対応中
- 修正完了時: 対応済
- 再レビュー完了後: クローズ
- 対応不要判断時: 見送り

## 実装レビュー観点

### 1. 一貫性
- 同種のクラスで実装方針が揃っているか
- 命名、戻り値、公開 API が統一されているか
- 正規化した値の扱いと保持値の扱いが一貫しているか

### 2. 不変条件・境界値
- コンストラクタや factory で不正状態を確実に防いでいるか
- `null`、空文字、最小値、最大値などの境界条件が適切に扱われているか
- `Optional`、`null`、特別値オブジェクトの使い分けが整理されているか

### 3. 依存と責務分離
- レイヤ間の依存方向が崩れていないか
- クラスが複数責務を持っていないか
- domain に framework、I/O、外部ライブラリの詳細が漏れていないか

### 4. 例外とエラーハンドリング
- 例外の種類、粒度、責務が一貫しているか
- `IllegalArgumentException` と独自例外の使い分けに方針があるか
- エラー情報が呼び出し側にとって扱いやすい形になっているか

### 5. セキュアな実装
- 機微情報を `toString`、例外、ログ、レスポンスに露出していないか
- 入力値検証がバイパスされない構造になっているか
- パストラバーサル、制御文字、危険な文字列などの防御が必要箇所で行われているか

### 6. 保守性
- 未使用コードや仕様残骸が残っていないか
- コメントと実装が一致しているか
- 時刻や ID 生成など、テストしづらい要素が直書きされていないか
- ローカル変数、引数、フィールドが再代入不要なら `final` で宣言されているか
- `final` を付けていても、参照先が可変オブジェクトの場合は真の不変性を誤認していないか
