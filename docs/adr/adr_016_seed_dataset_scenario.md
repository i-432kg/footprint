# ADR: seed データは件数指定ではなく固定シナリオで投入する

## ステータス
Accepted

## 背景
従来の seed データ投入は、`activeUserCount` / `inactiveUserCount` のような件数指定をもとにユーザー・投稿・返信を生成していた。

この方式には以下の問題があった。

- 何の状態を保証する seed なのかが件数から読み取りにくい
- 偶数投稿への返信や画像の順次割り当てなど、初期実装由来のルールが残っている
- 開発や確認で必要なのは「状態の種類」であり、「大量の件数」ではない
- 期待する状態を再現するために件数や画像数へ依存してしまう

今回必要な seed は、次の状態を明示的に用意できれば十分である。

- 投稿ありユーザー
- 投稿なしユーザー
- 返信ありユーザー
- 返信なしユーザー
- 返信あり投稿
- 返信なし投稿
- 返信あり返信
- 返信なし返信

## 決定
1. seed データ投入は件数指定方式をやめ、固定シナリオ方式へ切り替える
2. local / stg ともに、同じ状態を表す最小シナリオを投入する
3. シナリオは以下の最小構成とする
   - `poster_with_replies`: 投稿あり・返信ありユーザー
   - `poster_without_replies`: 投稿あり・返信なしユーザー
   - `non_poster_replier`: 投稿なし・返信ありユーザー
   - `post_with_replies`: 返信あり投稿
   - `post_without_replies`: 返信なし投稿
   - `reply_with_children`: 子返信を持つ返信
   - `reply_without_children`: 子返信を持たない返信
4. 画像は 2 投稿分を使用するが、manifest のエントリが 1 件しかない場合は同じ画像を再利用してよい
5. manifest エントリが 0 件の場合は、必要なシナリオを作れないため seed を失敗させる

## 理由

- seed の目的を「件数生成」から「状態保証」へ明確に変えられる
- 開発・確認時に必要なデータが何かをコードから直接読める
- 画像数や偶然の割り当て順に依存せず、必要な状態を安定して再現できる
- local / stg 間で seed の意味を揃えやすい

## 運用ルール

- 新しい seed 要件は、まず「必要な状態」を定義してからシナリオへ追加する
- `activeUserCount` / `inactiveUserCount` のような件数プロパティは追加しない
- seed 用 caption / reply message / email は安定した識別子を含む固定命名にする
- 子返信が必要な場合は、まず親返信を作成し、その返信 ID を再取得してから子返信を作成する

## この方針で避けるもの

- 状態保証と関係のない件数調整
- 画像数や投稿番号に依存した暗黙ルール
- seed の意図が読み取りにくいデータ生成
