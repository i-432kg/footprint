# ADR: ULID 系 ID 値オブジェクトは正規化せず厳密検証する

## ステータス
Accepted

## 対象
- `UserId`
- `PostId`
- `ReplyId`
- `ImageId`

## 背景
ULID 系 ID 値オブジェクトの実装で、入力を `trim()` して検証するものと、正規化後の値を保持するものが混在していた。

この状態だと、以下のような不整合が起きうる。

- 検証に使った値と内部保持値が一致しない
- 同じ見た目の ID でも、先頭末尾の空白付き入力が内部に残る
- 等価判定、検索、ログ、永続化時の挙動が不安定になる

## 検討した選択肢

### 1. 正規化を行う
例: `trim()` してから検証し、その正規化後の値を保持する

メリット
- API 入力や外部入力の表記ゆれを吸収できる
- 呼び出し側の扱いが少し楽になる
- 比較や永続化の前に内部表現を揃えやすい

デメリット
- 識別子入力の誤りを静かに吸収してしまう
- 「どこまで正規化するか」のルールが別途必要になる
- ID のような厳密一致対象では、補正自体が不自然になりやすい

### 2. 正規化を行わず厳密検証する
例: 入力値をそのまま検証し、空白や表記ゆれを含むものは reject する

メリット
- 入力と保持値が常に一致する
- 実装が単純で説明しやすい
- 識別子の厳密一致と相性が良い
- ドメイン層で曖昧さを持ち込まない

デメリット
- 呼び出し側で入力整形が必要になる場合がある
- presentation / application 境界で UX を考慮した前処理が必要になることがある

## 決定
`UserId` `PostId` `ReplyId` `ImageId` は、正規化を行わず、入力をそのまま厳密検証する。

具体的には以下をルールとする。

- `null` は reject する
- 空文字・空白のみは reject する
- ULID パターンに完全一致しないものは reject する
- `trim()` や大文字小文字変換などの補正は行わない
- 内部保持値は必ず入力値そのものと一致させる

## 理由
- ULID はユーザー入力テキストではなく、識別子である
- ドメイン層では曖昧さを吸収せず、厳密な値として扱う方が自然
- 「検証用の値」と「保持する値」を分けない方が不具合を防ぎやすい
- 入力の利便性が必要なら、presentation / application の境界で明示的に整形すべきである

## 実施内容
- `UserId` / `PostId` / `ReplyId` / `ImageId` から `trim()` ベースの検証を除去
- 空白付き入力を reject するテストを追加

## 今後の運用
- ULID 系 ID 値オブジェクトでは入力補正を行わない
- 境界で整形が必要な場合は、controller / request mapper / application service で明示的に実施する
- メールアドレスや検索キーワードのようなユーザー入力文字列とは、方針を分けて扱う
