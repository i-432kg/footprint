# ADR: 再代入しない変数は `final` で宣言する

## ステータス
Accepted

## 背景
本プロジェクトでは、値オブジェクトやイミュータブルなドメインモデルを中心に設計している。

一方で、メソッド内のローカル変数や引数、フィールドが再代入可能なままだと、以下の問題が起きやすい。

- 状態変化を追いづらくなる
- 途中代入によるバグが入りやすくなる
- 「不変に扱いたい」という設計意図がコード上で伝わりにくい

そのため、再代入しない変数は `final` で宣言する方針を採用する。

## 決定
- フィールドは原則 `final` とする
- メソッド引数は原則 `final` とする
- ローカル変数は、再代入しない場合 `final` とする

## 目的
- 再代入を防ぎ、コードの可読性を上げる
- 状態変化を最小化し、バグ混入を防ぐ
- ドメイン層のイミュータブル設計と実装スタイルを揃える
- レビュー時に「この値は変わらない」という前提を明確にする

## 補足
- `final` は参照の再代入を防ぐものであり、参照先オブジェクトの可変性までは保証しない
- `final List<X>` のような可変コレクションは、必要に応じて defensive copy や unmodifiable 化を検討する
- framework 都合で setter や可変フィールドが必要な DTO / Entity 相当には、この方針を機械的に適用しない

## 運用ルール
- 一時変数であっても、再代入しないなら `final` を付ける
- 再代入が必要な場合だけ `final` を外す
- 「後で代入するかもしれない」という理由で `final` を外さない
- 可変オブジェクトを扱う場合は、`final` の有無とは別に不変性を確認する

## この方針のメリット
- 再代入の有無が一目で分かる
- 手続き的な書き換えを抑制できる
- IDE の補助なしでも読みやすい
- 値オブジェクト中心のコードと相性が良い

## この方針のデメリット
- Java では記述量が少し増える
- 短いメソッドでは冗長に見えることがある
- `final` を付けても参照先の可変性は別問題である

## このプロジェクトでの扱い
- `domain` / `application` では強めに適用する
- `infrastructure` / `presentation` でも基本は同様とする
- ただし、フレームワークの制約を受ける request DTO やバインド用オブジェクトでは例外を認める
