# UT実装指針

## 基本方針

- UT は公開仕様に基づく振る舞いの検証を基本とする。
- 必要に応じて、分岐・例外・状態遷移などの観点を補完する。
- テストケースは、正常系・異常系・境界値・エッジケースに分類して記載する。
- 同値分割・境界値分析を用いて、入力条件と期待結果を明確にする。
- 重要ロジック、分岐の多い処理、障害影響の大きい処理を優先して設計・実装する。
- `docs/ut` 配下に対象パッケージと同じ構成のディレクトリを作成し、`ファイル名_test_design.md` を必ず作成する。
- テスト仕様書には少なくとも、対象、観点、正常系、異常系、境界値・エッジケースを記載する。
- AI で UT を自動作成する場合も、先にテスト仕様書を作成し、その内容に基づいて `test` パッケージ配下に UT を実装する。
- テストを実行し、カバレッジを確認する。
- カバレッジは不足箇所の把握に利用し、数値のみで品質を判断しない。

## コーディングルール

- テスト名は `should_期待動作_when_条件` を基本とする。
- 各テストメソッドには `@DisplayName` を付与し、日本語でテスト内容を見出し的に記述する。
- 1 テスト 1 観点を原則とし、失敗時に原因が追いやすい粒度で記述する。
- テストコードは Arrange / Act / Assert の構造で記述する。
- 実装詳細ではなく、入力・出力・副作用を検証する。
- private メソッドは直接テストせず、公開 API を通じて振る舞いを確認する。
- 現在時刻、乱数、DB、外部 API、ネットワーク、ファイル I/O などの非決定要素に直接依存させない。
- 例外系では、例外型だけでなくメッセージ、エラーコード、副作用が発生していないことも確認する。
- テストデータの意図が分かるように、マジックナンバーの直書きを避け、定数や Builder/Factory を活用する。
